## 수업용 노트북: `CTP49906_avllm_molab.py`

`CTP49906_avllm_molab.py`는 위의 두 실험을 **강의용 워크스루 + 실시간 플레이그라운드**로
바꿔 놓은, 독립 실행 가능한 [marimo](https://marimo.io) 노트북입니다. [molab](https://marimo.io/molab)
(무료 호스팅 GPU 런타임)에서 열어, 우선 위에서 아래로 한 번 읽어 동작 원리를 파악한 뒤,
직접 *만져 보도록* — 영상, 프롬프트, 프레임 수, 그리고 무엇보다 **어텐션 녹아웃(attention
knockout)**을 바꿔 가며 — 설계되었습니다. 목표는 논문의 핵심 질문에 대한 감을 기르는
것입니다: **오디오-비주얼 LLM이 답할 때, 정말로 본 것과 들은 것을 모두 사용하는가?**

### 왜 필요한가

*"어떤 사람이 피아노를 치고 있다"* 같은 캡션은, 모델이 영상을 봤든, 소리를 들었든, 아니면
그냥 언어적 사전 지식(language prior)에 기댔든 똑같이 그럴듯해 보입니다. 이 노트북은 그런
경우들에 대한 가설을 세우고 시험할 두 가지 해석 가능성(interpretability) 프로브를
학생들에게 제공합니다.

- **Logit Lens** — 각 레이어의 **정규화 전 잔차 스트림(raw residual stream)**을 오디오
  토큰 위치에서 꺼내 `lm_head`를 바로 적용합니다. thinker의 최종 RMSNorm을 거치지 않으며,
  오디오 위치에는 보정된 다음-토큰 언어 모델 목적도 없습니다. 따라서 디코딩된 토큰은
  실제 중간 다음-토큰 예측이 아니라 진단용 프로브입니다.
- **Attention Knockout** — 선택한 레이어에서 source→target의 **직접 어텐션 엣지**만
  막고 다시 실행합니다. 모달리티 자체나 잔차 스트림의 정보를 제거하지 않으며, 간접 경로도
  남습니다. 따라서 결론은 이 특정 엣지 개입에 한정해야 합니다.

목표는 어떤 수치를 재현하는 것이 아니라, *모델을 내부에서 읽어 내는* 감각을 기르고
"X를 막으면 출력이 Y처럼 바뀔 것이다"라는 **반증 가능한(falsifiable)** 가설을 세우는
연습을 하는 것입니다.

### 실행 방법 (molab)

1. molab에서 노트북을 열고 헤더의 notebook-specs 버튼으로 **GPU를 연결**하세요
   (`cuda:0`을 사용하며, 3B thinker는 넉넉히 올라갑니다).
2. 셀을 위에서 아래로 실행하세요. 셋업 셀은 의존성을 커널에 pip로 설치하고, 작은 PyAV
   심(shim)으로 `torchvision.io.read_video`를 복원하며(최근 torchvision이 비디오
   디코더를 제거함), 실험 코드를 import할 수 있도록 이 저장소(`src/` + 샘플 영상)를
   클론합니다.
3. 첫 실행은 모델 가중치(~8 GB)를 내려받으므로 가장 느립니다. 이후 셀들은 이미 로드된
   모델을 재사용합니다.
4. Qwen 가중치는 노트북과 `precomputed/meta.json`에 기록된 불변 revision으로 고정됩니다.
   수업 배포 전에는 `REPO_REF`도 배포한 강좌 태그로 바꾸세요. 로컬 체크아웃에서 열면 그
   소스를 그대로 사용합니다.

CUDA 머신이라면 `uvx marimo edit CTP49906_avllm_molab.py`로 로컬에서도 실행할 수
있습니다 — `# /// script` 헤더가 호환되는 의존성 버전을 고정해 둡니다.

**GPU가 없다면? GPU 없는 재생.** molab의 GPU를 쓸 수 없을 때는 상단 설정 셀에서
`USE_PRECOMPUTED = True`로 바꾸세요: 가이드 데모의 모든 비대화형 그래프가
[`precomputed/`](precomputed/)에 커밋된 산물로부터 GPU도, 8 GB 다운로드도 없이
렌더링됩니다(각각 *"Saved course replay — no GPU"* 라벨이 붙습니다). 기본값은 `False`(라이브
모델)입니다. 대화형 플레이그라운드와 티처 포싱 섹션은 여전히 GPU가 필요하며, 이
모드에서 실행하면 시끄럽게 실패합니다. 산물 재생성은 CUDA 머신에서
`python scripts/generate_precompute.py`로 합니다.

### 녹아웃 규칙 읽는 법

모든 녹아웃은 튜플 **`(source, target, start_layer, end_layer)`**이며, 의미는 다음과
같습니다.

> *thinker 레이어 `[start_layer, end_layer)` 구간에서, `source` 타입 토큰이 `target`
> 타입 토큰에 어텐션하는 것을 금지한다.*

학생들이 자주 헷갈리는 두 가지를 먼저 짚고 갑니다.

- **방향성이 있습니다.** 어텐션은 source(보는 쪽 토큰)에서 target(읽히는 쪽 토큰)으로
  흐릅니다. `generated → video`(답변이 프레임을 볼 수 없음)와 `video → generated`는 완전히
  다른 개입입니다. 의미 있는 방향은 거의 항상 *뒤쪽 토큰이 앞쪽 토큰을 읽는* 것인데,
  모델이 인과 마스킹(causal masking)되어 있어 한 토큰은 자기 자신과 그 이전의 모든
  것에만 어텐션할 수 있기 때문입니다.
- **`end_layer`는 배타적(exclusive)입니다.** `[0, 36)`은 레이어 0부터 35까지, 즉 3B
  thinker에서는 전부를 의미합니다(노트북이 실제 레이어 수를 출력합니다). 창을 좁히면
  **레이어 대역 민감도**를 시험할 수 있지만, 그것만으로 융합 모듈의 위치가 특정되지는
  않습니다. 정보가 이미 간접 경로로 이동했거나 막히지 않은 엣지로 우회할 수 있습니다.

개입에 쓰는 여섯 가지 위치 토큰 타입(모두가 모달리티인 것은 아님):

| 타입 | 무엇인가 |
| --- | --- |
| `query_text` | 프롬프트 / 지시문의 텍스트. |
| `audio` | 영상의 **사운드트랙**에서 나온 토큰. |
| `video` | 샘플링된 **프레임**에서 나온 토큰. |
| `image` | 정지 이미지 토큰 — 영상 입력에는 없으므로 여기선 작동하지 않음. |
| `generated` | 자기회귀 디코딩 중에 모델이 **생성하는** 토큰(생성하는 동안에만 존재). |
| `answer` | 모델 자신의 캡션을 **티처 포싱으로 다시 입력**한 것 — 순전파 한 번으로 점수를 매길 수 있게. 티처 포싱 Δ log-lik 섹션이 사용하며, 토큰 id가 아니라 위치로 태그됨. |

### 녹아웃 쌍(pair) 카탈로그 (각각 무엇을 묻는가)

시도해 볼 만한 개입들입니다. 처음 세 개는 `source`가 `generated`이므로 **Attention
Knockout 셀**에 속하고(생성 중에 작동), 마지막 두 개는 **오디오 위치 logit lens**를 바꾸므로
**플레이그라운드**에 속합니다.

| 규칙 | 던지는 질문 | 출력 변화가 의미하는 것 |
| --- | --- | --- |
| `generated → video` | *이 레이어 대역에서 생성 토큰 쿼리가 비디오 키를 직접 읽지 못하면 무엇이 바뀌는가?* | 변화는 해당 엣지에 대한 민감도와 일치합니다. 시각적 근거를 주장하려면 대조군이 필요하며, 무변화는 중복·간접 전달 때문일 수도 있습니다. |
| `generated → audio` | *생성 토큰 쿼리가 오디오 키를 직접 읽지 못하면 무엇이 바뀌는가?* | 변화만으로 “진짜 듣기”가 입증되지는 않습니다. 무음/불일치 대조군과 우회 경로를 시험하세요. |
| `generated → query_text` | *생성 토큰이 지시문을 직접 다시 읽지 못하면 무엇이 바뀌는가?* | 이 엣지 의존성과 일치하는 변화인지 확인하되, 지시 정보가 이미 다른 곳에 인코딩됐을 가능성도 비교하세요. |
| `audio → video` | *오디오 위치 프로브 통계가 비디오 키로 향하는 직접 엣지에 민감한가?* | 다양성 변화는 설명해야 할 효과일 뿐, 시각→오디오 융합이나 의미 품질을 입증하지 않습니다. |
| `video → audio` | *비디오 토큰 쿼리가 오디오 키로 향하는 직접 엣지에 민감한가?* | 후속 변화는 교차 모달 가설을 제안하지만, 마스킹·재정규화 효과와 구분할 대조군이 필요합니다. |

> **왜 다양성 스코어보드에서는 `generated`가 아무 일도 하지 않는가.** 다양성 스코어보드는
> 프롬프트에 대한 순전파(forward pass) 1회만 돌리므로(자기회귀 디코딩 없음), 규칙이 작용할
> **`generated` 토큰이 없습니다.** 거기서 `generated → …` 규칙은 아무것도 막지 못하고 Δ가
> 평평합니다 — 시도하면 노트북이 경고를 띄웁니다. 오디오 위치 점수를 움직이려면 **source**를
> 실제로 존재하는 모달리티(`audio`, `video`, `query_text`)로 두세요.
>
> **답변이 source가 *될 수 있는* 곳: 티처 포싱.** 가장 직관적인 질문 — *"답변이 사운드트랙을
> 들을 수 없다면, 그래도 소리를 묘사하는가?"* — 에는 답변이 입력으로 존재해야 합니다.
> **티처 포싱 Δ log-우도** 섹션이 바로 그 일을 합니다: 캡션을 한 번 생성하고, **`answer`**로
> 태그해 다시 입력한 뒤, `answer → audio`(또는 `→ video`, `→ query_text`)를 연속적이고
> 결정론적인 **Δ log-lik = knockout − baseline**으로 점수화합니다 — 음수는 경로가 잘리자
> 모델이 자기 캡션에 더 낮은 확률을 할당했다는 뜻입니다. 이는 통제된 직접-엣지 개입
> 측정이지만 target 모달리티가 캡션 의미를 제공했다는 증명은 아닙니다. 여기서는 답변이
> 입력으로 존재하기 때문에 `answer` source가 의미를 갖습니다.

### 플레이그라운드 (직접 만지는 단계)

마지막 `🎛️` 섹션은 다양성 측정을 폼으로 감싸며 — **▶를 누르기 전에는 아무것도 실행되지
않습니다** — 이미 로드된 모델을 재사용하므로 반복이 빠르고 추가 VRAM이 필요 없습니다.
컨트롤:

- **Clip** — 두 폼 모두 **Default**, 짝지은 **Silent control**, **Upload** 중 하나를 명시적으로
  고릅니다. Upload를 고르면 파일이 필수이며 묵시적으로 기본 영상으로 돌아가지 않습니다.
  업로드는 경로 안전한 SHA-256 파일명으로 저장되고 같은 내용 해시로 캐시됩니다. 250 MB나
  120초·1080p·60 FPS 또는 디코딩 메모리 한도를 넘거나, 재생 시간이 불명확하거나,
  오디오/비디오 스트림 또는 디코딩 가능한 첫 프레임이 없으면 모델 실행 전에 거부됩니다.
- **Frames** — 2–32; 프레임이 많을수록 시각 맥락이 풍부해지고(그리고 느려짐).
- **Prompt** — 지시문; 소리 vs 시각 쪽으로 유도해 보세요.
- **Knockout 켜기/끄기**, 그다음 **단일 규칙** 드롭다운(source, target, 레이어 범위)을
  쓰거나, 여러 규칙을 `source,target,start,end` 형식으로 `;`로 구분해 넣는 **고급(advanced)**
  필드(예: `audio,video,0,36 ; audio,image,0,36`)를 씁니다. 고급 필드가 채워지면 그쪽이
  우선합니다.
- **Compare** — 녹아웃 없는 기준선도 함께 실행해, 스코어보드가 레이어별
  **Δ (녹아웃 − 기준)**를 보여 주도록 합니다.

스코어보드 읽는 법: 각 레이어는 **오디오 위치에서 디코딩한 서로 다른 argmax 프로브 토큰
수**로 점수가 매겨집니다. **음(−)의 Δ**는 녹아웃 뒤 서로 다른 프로브 토큰 수가 줄었다는
뜻일 뿐, 표현의 개선·악화나 모달리티 융합을 입증하지 않습니다. 평평한 Δ도 무민감성,
중복/간접 경로, 거친 측정치, 또는 (`generated` source처럼) 작동할 수 없는 개입 중 무엇인지
추가 대조군으로 구분해야 합니다.

### 학생을 위한 실험 제안

모든 실행을 [실험 워크시트](WORKSHEET.md)에 기록하세요 — ▶ **전에** 가설, 그다음 결과와
판정. 워크시트의 마지막 블록은 최종 프로젝트의 핵심인 효과와 대조군 질문입니다.

1. **같은 영상에서 시각 vs 청각.** 녹아웃 셀에서 *"see and hear"* 프롬프트로
   `generated → video`를 돌린 뒤 `generated → audio`를 돌려 보세요. 어느 녹아웃이 캡션을
   더 많이 바꾸나요? 이것은 해당 직접 엣지 집합에 대한 민감도를 어디까지 보여 주며,
   모달리티 의존성을 주장하려면 어떤 추가 대조군이 필요할까요?
2. **후보 교차 모달 대역 시험.** 플레이그라운드에서 `audio → video`를 `[0, 12)`,
   `[12, 24)`, `[24, 36)` 구간으로 돌려 보세요. 어느 대역에서 다양성이 가장 크게
   변하나요? 이것이 융합이라는 주장을 반증할 대조군이나 다른 엣지 규칙은 무엇인가요?
3. **프롬프트 유도.** 영상을 고정한 채 프롬프트를 *"what you hear"*와 *"what you see"*
   사이에서 바꿔 보세요. 어떤 녹아웃도 하기 전에, 오디오 위치가 이미 다르게 디코딩되나요?
4. **직접 영상 가져오기.** 오디오와 시각이 *서로 어긋나는* 영상(예: 무관한 영상 위에 얹은
   내레이션)을 업로드해, 모델이 어느 모달리티를 보고하는지 확인해 보세요.
5. **규칙 쌓기.** 고급 필드로 `audio → video`와 `audio → query_text`를 동시에 녹아웃해
   보세요 — 다양성 변화가 더 커지나요, 상쇄되나요, 아니면 방향이 뒤집히나요?
6. **시각 vs 청각, 정량화.** 🎯 티처 포싱 섹션에서 같은 클립·프롬프트로 `answer → audio`와
   `answer → video`를 돌려 보세요. 실험 1은 어느 녹아웃이 *캡션을 더 많이 바꾸는지* 물었다면,
   이것은 총 Δ와 **토큰당 평균** Δ log-우도를 묻습니다. 이진 diff와 연속 측정은 직접 엣지
   민감도에 대해 같은 답을 내며, 더 강한 모달리티 의존성 주장에는 무엇이 여전히 부족한가요?
7. **어느 레이어 대역에 점수가 민감한가.** `answer → audio`를 `[0, 12)`, `[12, 24)`,
   `[24, 36)` 구간으로 돌려 보세요. 어느 레이어 대역을 잘랐을 때 캡션이 믿음을 가장 많이
   잃나요? 실험 2와 비교한 뒤, 국소적인 융합 모듈을 가정하지 않는 대안 설명도 제시하세요.
8. **의미 있는 널.** `assets/02321_silent.mp4`에 `answer → audio`를 돌려 보세요 — 샘플
   클립과 같은 프레임이지만 사운드트랙이 디지털 무음입니다. 오디오 토큰은 여전히
   존재합니다. Δ가 0에 가까워질지 먼저 예측하고 시험하세요. 전처리·위치 효과·어텐션
   재정규화가 남으므로 무음은 보장된 널이 아닙니다. 두 클립이 비슷하다면 어떤 해석이
   반증되며, 다음 대조군은 무엇이어야 할까요?
